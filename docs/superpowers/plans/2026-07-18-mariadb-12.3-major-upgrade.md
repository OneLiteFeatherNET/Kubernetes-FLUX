# MariaDB Galera Major Upgrade (11.8.8 → 12.3.2) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Move the production `mariadb-galera` cluster (namespace `mariadb-galera`, 3-node Galera, managed by `mariadb-operator`) from `mariadb:11.8.8` to `mariadb:12.3.2`, both LTS releases, with a documented abort/rollback point before more than one node is touched.

**Architecture:** This is a GitOps repo — no staging cluster exists (`feather-core` is the single production cluster per `CLAUDE.md`), so the change is validated live, but safely, by leaning on `mariadb-operator`'s default `updateStrategy: ReplicasFirstPrimaryLast`. That strategy already rolls one Galera pod at a time and blocks on `GaleraReady` before touching the next — so node 0 rolling to 12.3.2 first, while nodes 1/2 stay on 11.8.8, is itself the canary step. We watch that first node closely and have a scripted rollback (revert the image tag in git, force-reconcile) ready before the operator proceeds further.

**Tech Stack:** Flux CD (Kustomize + HelmRelease), `mariadb-operator` (`k8s.mariadb.com/v1alpha1` `MariaDB` CRD), official `mariadb` Docker Hub image, SOPS-encrypted secrets (not touched by this plan).

## Global Constraints

- No source file other than `infrastructure/clusters/feather-core/configs/mariadb-galera/mariadb.yaml` changes — this is a single-field image bump, not a config redesign (YAGNI).
- Every push to `main` is live — Flux polls the `GitRepository` every 1m and the `configs` Kustomization every 10m. Do not push Task 2 until Task 1's backup/health checkpoint is green.
- Do not run `flux reconcile` in a loop (`CLAUDE.md` warning) — reconcile the source once per task, then watch status, don't re-trigger while `Reconciling`.
- `spec.galera.recovery.minClusterSize` stays at `1` (already set, `mariadb.yaml:48`) — do not raise it "to be safe"; upstream docs say a higher value causes recovery to needlessly restart healthy replicas, which would misfire during the rolling update.
- Abort criterion (applies to every task from Task 2 onward): if any pod fails to reach `wsrep_local_state_comment = Synced` within 10 minutes of restart, or the `MariaDB` resource's `GaleraReady` condition flips to `False` and stays there past `clusterHealthyTimeout` (10m, `mariadb.yaml:49`), stop and execute the Task 2 rollback procedure before doing anything else.
- Known upstream facts baked into this plan (do not re-derive, just verify at runtime): the official `mariadb:12.3.2` image bundles the Galera wsrep provider out of the box (MariaDB split it into the `mariadb-server-galera` package starting at 12.3, and the Docker image Dockerfile was already updated to install it — this only matters for host-package installs, not this image); `mariadb-operator`'s CRD defaults `spec.galera.galeraLibPath` when unset (as configured here), so no CRD field changes are needed for the provider library path; a direct 11.8 → 12.3 upgrade is MariaDB's officially documented supported path (no need to step through 12.0/12.1/12.2); our `myCnf` (`mariadb.yaml:57-83`) uses none of the variables removed by 12.0/12.3 (`big_tables`, `storage_engine`, `large_page_size` — we use `default_storage_engine`, which is unaffected); GTID-replication carryover caveats in the 12.3 release notes don't apply — this is a Galera multi-master cluster, not master/replica.
- Residual, un-verifiable-in-advance risk: `mariadb-operator`'s own docs list MariaDB compatibility as an open-ended floor ("Community >= 10.6") with no explicit 12.x certification, and the operator's own reference/test image is still `mariadb:11.8.8`. This plan's whole structure (single-node canary + hard abort criterion + rollback) exists specifically to contain that risk.

### Task 1: Pre-upgrade health baseline and backup checkpoint

**Files:** None — pure verification, no repo changes.

**Interfaces:**
- Produces: a recorded baseline (`wsrep_cluster_size`, `wsrep_local_state_comment`, latest successful `PhysicalBackup` name) that Task 4 compares against.

- [ ] **Step 1: Confirm the MariaDB resource is fully healthy before touching anything**

Run:
```bash
kubectl get mariadb mariadb-galera -n mariadb-galera -o jsonpath='{range .status.conditions[*]}{.type}={.status}{"  "}{end}{"\n"}'
```
Expected: `Ready=True  GaleraInitialized=True  Updated=True  GaleraReady=True  GaleraConfigured=True` (all `True`). If anything is `False`, stop — do not start a major upgrade on an already-unhealthy cluster.

- [ ] **Step 2: Record per-pod Galera state**

Run for each of `mariadb-galera-0`, `mariadb-galera-1`, `mariadb-galera-2`:
```bash
for p in mariadb-galera-0 mariadb-galera-1 mariadb-galera-2; do
  echo "== $p =="
  kubectl exec -n mariadb-galera "$p" -c mariadb -- \
    mariadb -uroot -p"$(kubectl get secret mariadb -n mariadb-galera -o jsonpath='{.data.password}' | base64 -d)" \
    -N -e "SHOW STATUS LIKE 'wsrep_cluster_size'; SHOW STATUS LIKE 'wsrep_local_state_comment';"
done
```
Expected: `wsrep_cluster_size = 3` and `wsrep_local_state_comment = Synced` on all three pods. Write these down (or paste into the task-tracking notes) — this is the baseline Task 4 must match or exceed.

- [ ] **Step 3: Confirm the most recent scheduled backup succeeded, or trigger one**

`mariadb-galera-backup` is a `PhysicalBackup` custom resource (`infrastructure/clusters/feather-core/configs/mariadb-galera/phsysical-backup.yaml`), not a native Kubernetes CronJob — the operator itself spawns Jobs named `mariadb-galera-backup-<timestamp>` on its `0 */6 * * *` cron. Check the CR's own status first:
```bash
kubectl get physicalbackup mariadb-galera-backup -n mariadb-galera
kubectl get jobs -n mariadb-galera --sort-by=.status.completionTime | tail -5
```
Expected: `PhysicalBackup` shows `COMPLETE=True STATUS=Success`, and the most recent `mariadb-galera-backup-*` Job shows `COMPLETIONS 1/1` and completed within the last 6h. If it's older than 6h or failed, force an on-demand run by patching `spec.schedule.onDemand` to a new value (the operator triggers a fresh backup whenever this differs from `status.lastScheduleOnDemand` — there is no `immediate`-field retrigger or CronJob to hook into for an already-existing resource):
```bash
kubectl patch physicalbackup mariadb-galera-backup -n mariadb-galera \
  --type merge -p '{"spec":{"schedule":{"onDemand":"preupgrade-'"$(date +%s)"'"}}}'
kubectl wait -n mariadb-galera physicalbackup/mariadb-galera-backup --for=condition=complete --timeout=15m
```

- [ ] **Step 4: No commit** — this task only reads cluster state, nothing to commit.

### Task 2: Bump the image and watch the first (canary) pod

**Files:**
- Modify: `infrastructure/clusters/feather-core/configs/mariadb-galera/mariadb.yaml:9`

**Interfaces:**
- Consumes: the healthy baseline recorded in Task 1, Step 2.
- Produces: `mariadb-galera-0` (or whichever pod the operator restarts first) running `mariadb:12.3.2` and rejoined as `Synced`, OR a confirmed rollback if it fails.

- [ ] **Step 1: Change the image tag**

Edit `infrastructure/clusters/feather-core/configs/mariadb-galera/mariadb.yaml:9`:
```yaml
  image: "mariadb:12.3.2"
```
(only this line changes — same as the earlier 11.8.5 → 11.8.8 patch bump).

- [ ] **Step 2: Validate the Kustomize build locally before pushing**

Run:
```bash
kubectl kustomize infrastructure/clusters/feather-core/configs/mariadb-galera | grep "image:"
```
Expected: `image: mariadb:12.3.2`

- [ ] **Step 3: Commit**

```bash
git add infrastructure/clusters/feather-core/configs/mariadb-galera/mariadb.yaml
git commit -m "$(cat <<'EOF'
fix(mariadb): major upgrade to 12.3.2 LTS

Moves the Galera cluster from the 11.8 LTS line to the 12.3 LTS line
(support through 2029). mariadb-operator's default
ReplicasFirstPrimaryLast update strategy rolls one replica at a time
and holds the primary until last, giving a single-node canary window
before the rest of the cluster follows.
EOF
)"
```

- [ ] **Step 4: Push to main and reconcile once**

```bash
git push origin HEAD:main
flux reconcile source git flux-system
flux reconcile kustomization configs --with-source
```
Do not re-run `flux reconcile` again in this task — after this, only watch (Step 5).

- [ ] **Step 5: Watch the first pod's rollout closely — this is the canary**

```bash
kubectl get mariadb mariadb-galera -n mariadb-galera -w
```
Expected sequence: `STATUS` moves `Running` → `Pending update` → `Updating` → `Running`. In a second terminal/command, watch the pod itself:
```bash
kubectl get pods -n mariadb-galera -l app.kubernetes.io/instance=mariadb-galera -w
```
The first pod to restart will show `Terminating` → `Init` → `Running` with `READY 0/1` → `1/1`. Once it reports `1/1` ready, confirm it actually rejoined Galera (readiness alone isn't proof — check wsrep state directly, and check the container actually came up as 12.3.2):
```bash
FIRST_POD=$(kubectl get pods -n mariadb-galera -l app.kubernetes.io/instance=mariadb-galera \
  -o jsonpath='{range .items[*]}{.metadata.name}{" "}{.spec.containers[?(@.name=="mariadb")].image}{"\n"}{end}' \
  | grep 12.3.2 | head -1 | cut -d' ' -f1)
echo "canary pod: $FIRST_POD"
kubectl exec -n mariadb-galera "$FIRST_POD" -c mariadb -- \
  mariadb -uroot -p"$(kubectl get secret mariadb -n mariadb-galera -o jsonpath='{.data.password}' | base64 -d)" \
  -N -e "SELECT VERSION(); SHOW STATUS LIKE 'wsrep_cluster_size'; SHOW STATUS LIKE 'wsrep_local_state_comment';"
```
Expected: `VERSION()` starts with `12.3.2`, `wsrep_cluster_size = 3`, `wsrep_local_state_comment = Synced`.

- [ ] **Step 6: Abort/rollback procedure (only if Step 5's expectations are not met within 10 minutes)**

```bash
git revert --no-edit HEAD
git push origin HEAD:main
flux reconcile source git flux-system
flux reconcile kustomization configs --with-source
kubectl get mariadb mariadb-galera -n mariadb-galera -w
```
Watch until all three pods are back on `11.8.8` and `GaleraReady=True`. Stop here, do not proceed to Task 3, and report the failure signal (pod logs, `wsrep` status, operator events) before attempting the upgrade again.

### Task 3: Let the remaining nodes (including primary switchover) complete

**Files:** None — no further repo changes, this task is monitoring only.

**Interfaces:**
- Consumes: canary pod confirmed healthy on `12.3.2` from Task 2.
- Produces: all 3 pods on `12.3.2`, cluster `Ready`/`GaleraReady`, and a new primary recorded.

- [ ] **Step 1: Watch the operator finish the rollout unattended**

`ReplicasFirstPrimaryLast` will now roll the remaining replica, then switch over and restart the primary last, same sequence already observed during the 11.8.5→11.8.8 bump. Watch:
```bash
kubectl get mariadb mariadb-galera -n mariadb-galera -w
```
Expected: eventually `STATUS=Running`, `UPDATES=ReplicasFirstPrimaryLast`, and a `PRIMARY` value (may differ from before the upgrade — that's expected, the operator promotes a synced replica before restarting the old primary).

- [ ] **Step 2: Confirm all three pods are on the new version and synced**

```bash
kubectl get pods -n mariadb-galera -o jsonpath='{range .items[?(@.metadata.labels.app\.kubernetes\.io/instance=="mariadb-galera")]}{.metadata.name}{"\t"}{.spec.containers[?(@.name=="mariadb")].image}{"\n"}{end}'
for p in mariadb-galera-0 mariadb-galera-1 mariadb-galera-2; do
  echo "== $p =="
  kubectl exec -n mariadb-galera "$p" -c mariadb -- \
    mariadb -uroot -p"$(kubectl get secret mariadb -n mariadb-galera -o jsonpath='{.data.password}' | base64 -d)" \
    -N -e "SHOW STATUS LIKE 'wsrep_cluster_size'; SHOW STATUS LIKE 'wsrep_local_state_comment';"
done
```
Expected: all three pods show `mariadb:12.3.2`, `wsrep_cluster_size = 3`, `wsrep_local_state_comment = Synced` on every pod.

- [ ] **Step 3: No commit** — monitoring only, no repo changes in this task.

### Task 4: Post-upgrade validation

**Files:** None.

**Interfaces:**
- Consumes: fully-rolled 12.3.2 cluster from Task 3.
- Produces: confirmation that `mariadb-upgrade` ran, applications can still connect, metrics still scrape, and a fresh backup succeeds on the new version — the exit criteria for calling this upgrade done.

- [ ] **Step 1: Confirm `mariadb-upgrade` ran automatically**

The official image's entrypoint runs `mariadb-upgrade` automatically on first startup against an existing data directory from an older version. Confirm it actually happened rather than assuming:
```bash
kubectl logs -n mariadb-galera mariadb-galera-0 -c mariadb --since=1h | grep -i "upgrade"
```
Expected: log lines mentioning `mariadb-upgrade` running and completing (e.g. `mariadb-upgrade ... OK` / `Phase ... completed`). Repeat for `mariadb-galera-1` and `mariadb-galera-2`. If no such lines appear on a pod, run it manually:
```bash
kubectl exec -n mariadb-galera <pod> -c mariadb -- \
  mariadb-upgrade -uroot -p"$(kubectl get secret mariadb -n mariadb-galera -o jsonpath='{.data.password}' | base64 -d)"
```

- [ ] **Step 2: Spot-check application connectivity**

Pick two or three of the databases under `infrastructure/clusters/feather-core/configs/mariadb-galera/databases/` (e.g. `shlink`, `leantime`) and confirm their consuming app is still healthy post-upgrade:
```bash
kubectl get pods -A -l app.kubernetes.io/name=shlink
kubectl logs -n <shlink-namespace> <shlink-pod> --since=15m | tail -30
```
Expected: no new connection/auth errors since the rollout completed. If an app pod is crash-looping or logging DB errors, investigate before closing this out — do not roll it into "expected noise."

- [ ] **Step 3: Confirm metrics are still scraping**

```bash
kubectl get pods -n mariadb-galera -l app.kubernetes.io/name=mariadb-galera-metrics
kubectl logs -n mariadb-galera -l app.kubernetes.io/name=mariadb-galera-metrics --since=15m | tail -20
```
Expected: exporter pod `Running`, no repeated connection errors to `mariadb-galera` in its logs (`mariadb.yaml:131-148` wires this to the same cluster via the `monitoring` user).

- [ ] **Step 4: Confirm a fresh backup succeeds on the new version**

Force another on-demand run the same way as Task 1 Step 3, with a distinct `onDemand` value:
```bash
kubectl patch physicalbackup mariadb-galera-backup -n mariadb-galera \
  --type merge -p '{"spec":{"schedule":{"onDemand":"postupgrade-'"$(date +%s)"'"}}}'
kubectl wait -n mariadb-galera physicalbackup/mariadb-galera-backup --for=condition=complete --timeout=15m
```
Expected: `physicalbackup.k8s.mariadb.com/mariadb-galera-backup condition met`, and a new `mariadb-galera-backup-<timestamp>` Job with `COMPLETIONS 1/1`.

- [ ] **Step 5: No commit** — validation only. If Task 1–4 all pass, the upgrade is complete; nothing further to push.
